# 물건 히스토리 - 경우의 수 상세 문서

> 기준일: 2026-03-17
> 참조 파일: `js-app.html` (렌더링), `SheetDB.js` (저장), `TelegramService.js` (텔레그램 이력)

---

## 1. 히스토리 테이블 컬럼 구성

| 컬럼 | 표시 내용 | 설명 |
|------|----------|------|
| **#** | 행 번호 | 최신순 (1이 가장 최신) |
| **날자** | yyyy-MM-dd | 이벤트 발생 날짜 |
| **시간** | HH:mm ss | 이벤트 발생 시간 |
| **변경자** | trigger_type 값 | 누가/어떤 경로로 변경했는지 |
| **작업** | action + note | 어떤 작업인지 |
| **상태** | stu_member (물건상태) | 해당 시점의 물건상태 / 변경 전→후 표시 |
| **입찰가** | bidprice | 해당 시점의 입찰가 / 변경 전→후 표시 |
| **회원** | m_name (회원명) | 해당 시점의 회원명 / 변경 전→후 표시 |
| **담당자** | m_name_id (입찰담당) | 해당 시점의 담당자 / 변경 전→후 표시 |

> **표시 규칙**: 변경된 필드는 `이전값(회색 취소선)` + `변경값(빨간색 굵게)`, 미변경 필드는 검은색, 물건등록은 파란색 굵게

---

## 2. 변경자(trigger_type) 종류

| trigger_type 값 | 화면 표시 | 색상 | 발생 경로 |
|-----------------|-----------|------|----------|
| `web` | **Web** | 빨간색 굵게 | 웹 화면에서 수동 저장 (updateData) |
| `web-telegram` | **web-telegram** | 빨간색 굵게 | 웹에서 텔레그램 발송 버튼 클릭 |
| `web-다중` | **web-다중** | 빨간색 굵게 | 웹에서 일괄 처리 |
| `member-telegram` | **mem-telegram** | 파란색 굵게 | 회원이 텔레그램으로 응답 |
| `system` | sys | 회색 | 시스템 자동 처리 (기본값) |

## 3. 작업(action) 종류 및 발생 조건

### 3-1. FIELD_CHANGE — 필드 변경

**화면 표시**: `필드 변경`

#### 발생 조건 및 상세

| 변경 필드 | 한국어명 | 언제 발생 | trigger_type | 비고 |
|-----------|---------|-----------|--------------|------|
| `stu_member` | 물건상태 | 웹 저장 폼에서 물건상태 변경 시 | `web` | 추천→입찰, 미정→추천 등 |
| `m_name_id` | 입찰담당 | 웹 저장 폼에서 담당자 변경 시 | `web` | |
| `m_name` | 회원명 | 웹 저장 폼에서 회원 변경 시 | `web` | |
| `bidprice` | 입찰가 | 웹 저장 폼에서 입찰가 변경 시 | `web` | |
| `member_id` | 담당회원ID | 웹 저장 폼에서 회원ID 변경 시 | `web` | 화면 컬럼에는 표시 안 됨 |
| `bid_state` | 입찰가상태 | 웹 저장 폼에서 입찰가상태 변경 시, 또는 bid_state 일괄 업데이트 | `web` | 화면 컬럼에는 표시 안 됨 |
| `chuchen_state` | 추천상태 | 신규/전달완료 버튼 클릭 저장 | `web` | 화면 컬럼에는 표시 안 됨 |
| `chuchen_state` | 추천상태 | 텔레그램 추천 발송 후 자동 전달완료 처리 | `web-telegram` | 화면 컬럼에는 표시 안 됨 |

**그룹화 규칙**: 같은 분(minute) + 같은 action + 같은 trigger_type 이면 **한 행으로 합쳐서 표시**
→ 예: 동시에 stu_member + m_name + m_name_id 가 바뀌면 1개 행에 3개 컬럼 모두 변경 표시

#### 셀 표시 형식 (FIELD_CHANGE)

```
이전값이 있으면: (이전값 - 회색 취소선)
                  변경값 (빨간색 굵게)

이전값 없으면:   (없음 - 회색 취소선)
                  변경값 (빨간색 굵게)

값 동일:         변경값 (검은색 - 실제 변경 없음)
```

### 3-2. ITEM_CREATE — 물건 등록

**화면 표시**: `물건 등록`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 신규 물건 등록 시 |
| trigger_type | `web` |
| 셀 표시 | 등록 당시 값 → **파란색 굵게** |
| 비고 | 최초 등록 시 상태, 회원, 입찰가 등이 파란색으로 표시됨 |

---

### 3-3. ITEM_DELETE — 물건 삭제

**화면 표시**: `물건 삭제`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 물건 삭제 처리 시 |
| trigger_type | `web` |
| 비고 | 삭제된 시점의 스냅샷 표시 |

---

### 3-4. TELEGRAM_SENT — 텔레그램 발송

**화면 표시**: `텔레그램`
하단에 `(note 종류)` 2줄로 표시

#### note 종류별 상세

| note 값 | 화면 표시 (2번째 줄) | 발생 조건 | trigger_type |
|---------|-------------------|-----------|--------------|
| `card` | `(추천물건전달)` | 웹에서 추천물건 텔레그램 발송 버튼 클릭 | `web-telegram` |
| `bid_price` | `(입찰가전달)` | bid_state를 '전달완료'로 저장 시 자동 발송, 또는 직접 발송 | `web-telegram` |
| `status` | `(진행상태전달)` | 진행상태(변경/취소) 텔레그램 전달 | `web-telegram` |
| `check_request` | `(확인요청전달)` | 확인 요청 텔레그램 발송 | `web-telegram` |

---

### 3-5. TELEGRAM_RECEIVED — 텔레그램 수신

**화면 표시**: `텔레그램회신 (note 종류)`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 회원이 텔레그램으로 메시지 응답 시 |
| trigger_type | `member-telegram` → 화면에 **mem-telegram** (파란색) |
| note | 수신된 메시지 유형 |

---

### 3-6. REQUEST_BID — 입찰 요청

**화면 표시**: `입찰 요청`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 회원이 텔레그램에서 "입찰확정" 버튼 클릭 (PENDING 상태로 생성) |
| trigger_type | `member-telegram` → **mem-telegram** (파란색) |
| status | `PENDING` → 승인 처리 전 |
| 비고 | 승인 후 REQUEST_APPROVED + FIELD_CHANGE(stu_member→입찰) 이력 추가됨 |

---

### 3-7. REQUEST_CANCEL_CHUCHEN — 추천 취소 요청

**화면 표시**: `추천 취소 요청`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 회원이 텔레그램에서 "이 물건 선택 안함" 버튼 클릭 |
| trigger_type | `member-telegram` → **mem-telegram** (파란색) |
| status | `PENDING` |
| 비고 | 승인 후 FIELD_CHANGE(stu_member→미정 또는 상품) 이력 추가됨 |

---

### 3-8. REQUEST_CANCEL_BID — 입찰 취소 요청

**화면 표시**: `입찰 취소 요청`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 회원이 텔레그램에서 "입찰취소" 버튼 클릭 |
| trigger_type | `member-telegram` → **mem-telegram** (파란색) |
| status | `PENDING` |
| 비고 | 승인 후 FIELD_CHANGE(stu_member: 입찰→다른값) 이력 추가됨 |

---

### 3-9. REQUEST_APPROVED — 요청 승인

**화면 표시**: `요청 승인 (자동/수동)`

| 항목 | 내용 |
|------|------|
| 발생 조건 | PENDING 요청(REQUEST_BID, REQUEST_CANCEL_CHUCHEN, REQUEST_CANCEL_BID)이 승인 처리될 때 |
| trigger_type | 자동 승인: `system`, 수동 승인: `web` |
| 비고 | 승인과 동시에 FIELD_CHANGE 이력도 함께 기록됨 |

---

### 3-10. REQUEST_REJECTED — 요청 반려

**화면 표시**: `요청 반려`

| 항목 | 내용 |
|------|------|
| 발생 조건 | PENDING 요청이 반려 처리될 때 |
| trigger_type | `web` |

---

### 3-11. EXPIRY_NOTIFY — 만료 예정 알림

**화면 표시**: `만료 예정 알림`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 추천 물건이 만료 예정일 때 알림 발송 (24h, 1h 전) |
| trigger_type | `system` → sys (회색) |
| note | `24h` 또는 `1h` |

---

### 3-12. AUTO_EXPIRE — 자동 추천 만료

**화면 표시**: `자동 추천 만료`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 추천 후 48시간 경과하여 자동으로 미정 전환될 때 |
| trigger_type | `system` → sys (회색) |
| 비고 | AUTO_EXPIRE_ENABLED 설정이 true일 때만 동작 |

---

### 3-13. BID_DATE_NOTIFY — 입찰일 알림

**화면 표시**: `입찰일 알림`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 입찰일 D-3, D-2, D-1 알림 발송 시 |
| trigger_type | `system` → sys (회색) |
| note | `d3`, `d2`, `d1` |

---

### 3-14. PRICE_CONFIRMED — 입찰가 확인

**화면 표시**: `입찰가확인`

| 항목 | 내용 |
|------|------|
| 발생 조건 | 회원이 텔레그램에서 입찰가 "확인" 버튼 클릭 시 |
| trigger_type | `member-telegram` → **mem-telegram** (파란색) |
| field_name | `bid_state` |
| to_value | `확인완료` |
| 비고 | bid_state는 화면 컬럼에 표시 안 됨 (내부 필드) |

## 4. 전체 시나리오별 히스토리 흐름

### 시나리오 A: 추천물건 전달 흐름 (전형적인 케이스)

```
행 순서 (오래된→최신)
─────────────────────────────────────────────────────
1. FIELD_CHANGE   | Web         | 상태: 미정→추천, 회원: 홍길동 배정
2. TELEGRAM_SENT  | web-telegram| (추천물건전달)
3. FIELD_CHANGE   | web-telegram| chuchen_state: 신규→전달완료  ← 화면 컬럼 미표시
─────────────────────────────────────────────────────
```

### 시나리오 B: 입찰 확정 흐름

```
─────────────────────────────────────────────────────
1. (추천 후 위 A 시나리오)
2. REQUEST_BID         | mem-telegram | 회원 입찰확정 요청 (PENDING)
3. REQUEST_APPROVED    | system/web   | 승인 처리
4. FIELD_CHANGE        | system/web   | 상태: 추천→입찰
─────────────────────────────────────────────────────
```

### 시나리오 C: 추천 취소(미선택) 흐름

```
─────────────────────────────────────────────────────
1. REQUEST_CANCEL_CHUCHEN | mem-telegram | 미선택 요청 (PENDING)
2. REQUEST_APPROVED       | system/web   | 승인
3. FIELD_CHANGE           | system/web   | 상태: 추천→미정, 회원: 홍길동→(없음)
─────────────────────────────────────────────────────
```

### 시나리오 D: 입찰가 전달 및 확인 흐름

```
─────────────────────────────────────────────────────
1. FIELD_CHANGE        | Web         | bidprice: (없음)→77990000, bid_state: 신규→전달완료
2. TELEGRAM_SENT       | web-telegram| (입찰가전달)   ← bid_state=전달완료 저장 시 자동 발송
3. PRICE_CONFIRMED     | mem-telegram| 회원이 확인 버튼 클릭
4. FIELD_CHANGE        | system      | bid_state: 전달완료→확인완료  ← 화면 컬럼 미표시
─────────────────────────────────────────────────────
```

### 시나리오 E: 입찰 취소 흐름

```
─────────────────────────────────────────────────────
1. REQUEST_CANCEL_BID  | mem-telegram | 입찰취소 요청 (PENDING)
2. REQUEST_APPROVED    | system/web   | 승인
3. FIELD_CHANGE        | system/web   | 상태: 입찰→미정 (또는 상품)
─────────────────────────────────────────────────────
```

### 시나리오 F: 변경/취소 안내 흐름

```
─────────────────────────────────────────────────────
1. FIELD_CHANGE        | Web          | 상태: 입찰→변경
2. TELEGRAM_SENT       | web-telegram | (진행상태전달)
─────────────────────────────────────────────────────
```

### 시나리오 G: 자동 만료 흐름

```
─────────────────────────────────────────────────────
1. EXPIRY_NOTIFY  | system | 만료 예정 알림 (24h 전)
2. EXPIRY_NOTIFY  | system | 만료 예정 알림 (1h 전)
3. AUTO_EXPIRE    | system | 자동 만료 처리
4. FIELD_CHANGE   | system | 상태: 추천→미정
─────────────────────────────────────────────────────
```

## 5. 화면에 표시되는 필드 vs 내부 추적 전용 필드

| 필드 | 화면 컬럼 표시 | FIELD_CHANGE 추적 | 비고 |
|------|--------------|-----------------|------|
| stu_member (물건상태) | ✅ 상태 | ✅ | |
| bidprice (입찰가) | ✅ 입찰가 | ✅ | |
| m_name (회원명) | ✅ 회원 | ✅ | |
| m_name_id (입찰담당) | ✅ 담당자 | ✅ | |
| member_id (담당회원ID) | ❌ | ✅ | 화면에는 안 보임 |
| bid_state (입찰가상태) | ❌ | ✅ | 화면에는 안 보임 |
| chuchen_state (추천상태) | ❌ | ✅ | 화면에는 안 보임 |

---

## 6. 그룹화 규칙 (행 합치기)

같은 **분(minute) + action + trigger_type** 이 동일하면 **1개 행**으로 합쳐서 표시

**예시**: 저장 시 stu_member, m_name, m_name_id 동시 변경
→ FIELD_CHANGE 3건이 기록되지만 화면에는 **1개 행**에 3개 컬럼 모두 변경 표시

## 7. 저장 위치

모든 이력은 `telegram_requests` 시트에 저장됨

| 열 | 필드명 | 내용 |
|----|--------|------|
| A | req_id | 타임스탬프 기반 고유ID |
| B | requested_at | 이벤트 발생 일시 |
| C | action | 작업 종류 |
| D | status | PENDING / DONE |
| E | item_id | 물건 ID |
| F | member_id | 회원 ID |
| G | chat_id | 텔레그램 chat_id |
| H | telegram_username | 텔레그램 사용자명 |
| I | note | 메모 (card, bid_price, status 등) |
| J | approved_at | 처리 일시 |
| K | approved_by | 처리자 |
| L | from_value | 변경 전 값 |
| M | to_value | 변경 후 값 |
| N | field_name | 변경된 필드명 |
| O | trigger_type | 트리거 종류 |
| P | member_name | 이벤트 시점 회원명 |
